# WASA調整君 要件定義書 (docs/spec.md)

最終更新：2026-07-11
ステータス：MVP要件確定

---

## 1. 概要

### 1.1 目的
WASA(早稲田大学宇宙航空研究会)の班ミーティング調整用の日程調整Webアプリ．
既存アプリ(タッピー・トントン等)のバグ・入力の煩雑さを解消し，自分たちの要望を直接反映できるツールを自作する．

### 1.2 コンセプト
- **アカウント登録不要**：URLを共有するだけで誰でも回答できる
- **一括入力**：ドラッグで塗るように◯△×を入力できる
- **完全無料運用**：無料枠のみで構築・運用する

### 1.3 想定規模
- 1イベントあたり回答者 20人程度，候補スロット 50個前後
- 利用シーン：WASAの班ミーティング調整

### 1.4 リリース目標
2026年夏休み中(8月末まで)に実運用開始．

## 2. 機能要件

### 2.1 イベント作成

- 入力項目：
  - イベント名(必須)
  - 説明文(任意)
  - 管理用パスワード(必須，主催者機能に使用)
  - 候補スロットの定義(2.2参照)
- 作成完了時に以下を発行・表示する：
  - 共有用URL：`/e/{eventId}`(eventIdはnanoid 21文字，推測不可能)
  - 「URLをコピー」ボタン
- 作成後の編集：**候補スロットの追加のみ可**(削除は不可)．編集には管理用パスワードが必要．

### 2.2 候補スロットの形式

3形式をサポートする．データモデル上はすべて「スロット＝ラベル+ソート順」に正規化する(6.2参照)．

| 優先度 | 形式 | 内容 |
|---|---|---|
| P1 (MVP) | 日付+時間指定形式 | 日付範囲 × 時間範囲 × 刻み幅からグリッドを一括生成 |
| P2 | 時間割形式 | 曜日(月〜日から選択) × 時限(1限〜7限，汎用ラベル) |
| P3 | 自由ラベル形式 | 「7/20午後」「未定」等の手入力ラベル |

**日付+時間指定形式の詳細(P1)：**
- 日付範囲指定(例：7/20〜7/26)+ 時間範囲指定(例：9:00〜18:00)で一括生成できること
- 刻み幅は**自由入力**(分単位，例：10分/15分/30分/60分/90分)．バリデーション：5〜720分，時間範囲を割り切れない場合は最終スロットを短くするか警告を出す
- 特定の日付を生成後に除外できること(例：7/23だけ候補から外す)

**時間割形式の詳細(P2)：**
- 汎用的な「1限〜7限」ラベルを使用(大学プリセットは持たない)
- 曜日は月〜日から複数選択

### 2.3 回答機能

#### 回答の選択肢
- ◯ / △ / × の3値
- **未入力のスロットは × とみなす**(集計時に×として扱う．DBには◯と△のみ保存し，レコードが無いスロット＝×)

#### 回答フロー
1. 共有URLを開く
2. 名前(表示名)と**回答用パスワード**を入力
3. グリッドで◯△×を入力
4. 送信

#### 回答の編集(パスワード方式)
- 回答者は自分で設定したパスワードを入力すれば**どの端末からでも**自分の回答を編集できる
- パスワードはハッシュ化して保存(bcrypt)．平文保存は禁止
- パスワードを忘れた場合：主催者が管理用パスワードでその回答を**削除**できる(編集は不可)→ 本人が再回答
- 補助として，回答成功時にlocalStorageへ「イベントID+回答者ID」を保存し，同一端末からの再訪時は名前を自動入力する(パスワードは保存しない)

#### 同名回答者の扱い
- 同名でも**別人として両方表示**する(回答者はIDで区別)
- 主催者は管理用パスワードで任意の回答を削除できる

### 2.4 入力UX(本アプリの核心)

#### モード切替方式の一括入力
- 画面下部(スマホ)/上部(PC)に「◯ / △ / ×」のモード選択ボタンを固定表示
- 選択中のモードで，セルをタップ or **ドラッグ(スワイプ)でなぞった範囲を一括塗り**できる
- 実装要件：
  - Pointer Events(`pointerdown`/`pointermove`/`pointerup`)で実装し，マウス・タッチ両対応
  - グリッド上では`touch-action: none`を適用し，ドラッグ中の画面スクロールを抑制する
  - グリッド外のスクロール操作は妨げない(グリッドが画面より大きい場合のスクロール手段を確保：スクロール専用ハンドル or 2本指スクロール)

#### 行・列の一括選択
- 行ヘッダー(日付)タップ → その行全体を現在のモードで塗る
- 列ヘッダー(時刻)タップ → その列全体を現在のモードで塗る

#### グリッドの表示
- 縦軸：時刻，横軸：日付(調整さん等と同じ向き)．50スロット×7日程度が想定上限
- スマホでは横スクロール可能なグリッド+固定ヘッダー(sticky)

### 2.5 集計・表示

- **ヒートマップ表示**：◯=2点，△=1点，×=0点(未入力含む)で各スロットのスコアを計算し，スコアに応じた色の濃淡で表示する
  - 色は連続的なグラデーション(例：白→緑)
  - 満点(全員◯)のスロットは強調表示(枠線等)
- セル内にスコアまたは「◯人数」を小さく併記する
- 回答者一覧を表示する(回答者名のリスト)
- リアルタイム反映は行わない．**リロード/更新ボタンで再取得**する方式
- (優先度：低，Phase 3で余裕があれば)スロットタップで「誰が◯/△/×か」の内訳をモーダル表示

### 2.6 主催者機能

- 識別方式：イベント作成時に設定する**管理用パスワード**
- できること：
  - 候補スロットの追加
  - 任意の回答の削除
  - イベント自体の削除
- 管理操作の入口：イベントページ内の「主催者メニュー」からパスワードを入力
- 管理用パスワードもbcryptでハッシュ化保存
- 締切機能・決定日時の確定機能は**実装しない**(明示的にスコープ外)

### 2.7 イベント一覧ページ(マイページ相当)

- localStorageに「作成したイベント」「回答したイベント」のIDと名前を保存し，一覧表示するページ `/mypage` を用意する
- サーバー側にはユーザー概念を持たない(あくまでlocalStorageベース)
- LINE内ブラウザ等でlocalStorageが利用できない場合はその旨を表示し，機能をグレースフルに無効化する

## 4. 非機能要件

### 4.1 対応環境
| 環境 | 対応 |
|---|---|
| スマホ LINE内ブラウザ | **必須**(最重要，URLはLINEで共有される) |
| スマホ Safari / Chrome | 必須 |
| PC Chrome / Edge | 必須 |
| PC Firefox / Safari | 必須 |

- モバイルファーストで設計する(回答の大半はスマホから)
- LINE内ブラウザ固有の注意：localStorageの永続性が保証されない前提で設計する(2.7参照)

### 4.2 プライバシー・セキュリティ
- 収集する個人情報は**表示名のみ**
- 全ページに`noindex`メタタグを付与(検索エンジンに載せない)
- イベントIDはnanoid(21文字)で推測不可能にする
- パスワードは全てbcryptハッシュで保存
- DBへの書き込みは全てNext.jsのAPI Route(サーバー側)経由で行い，Supabaseのservice roleキーはサーバーにのみ置く．クライアントから直接書き込ませない
- XSS対策：ユーザー入力(イベント名，回答者名等)は必ずエスケープして表示(Reactのデフォルト挙動を崩さない)

### 4.3 データ寿命
- **最終更新から6ヶ月経過したイベントを自動削除**する
- イベントページに「最終更新から6ヶ月で自動削除されます」の注意書きを表示

### 4.4 コスト
- 全て無料枠で運用する：
  - Vercel Hobbyプラン
  - Supabase Freeプラン
  - GitHub Actions(publicリポジトリなら無制限)
- **Supabase 7日停止対策**：GitHub Actionsで毎日1回，keep-alive用のクエリを実行する(4.5参照)

### 4.5 定期ジョブ(GitHub Actions，毎日1回)
1つのワークフローで以下を実行：
1. Supabaseへのkeep-aliveクエリ(一時停止防止)
2. 最終更新から6ヶ月超のイベントの削除(関連する回答・スロットもCASCADE削除)

### 4.6 品質・テスト
- **E2Eテスト(Playwright)をMVPから導入**する．対象は主要フロー：
  1. イベント作成 → 共有URL発行
  2. 回答入力(タップ+ドラッグ) → 送信 → 集計に反映
  3. パスワードによる回答編集
  4. 主催者パスワードによる回答削除
- CIでE2Eテストを実行(GitHub Actions)
- モバイルビューポートでのテストも含める(Playwrightのデバイスエミュレーション)

## 6. データモデル

### 6.1 ER概要
```
events 1 ─── n slots
events 1 ─── n respondents
respondents 1 ─── n answers (answerはslotに紐づく)
```

### 6.2 テーブル定義

```sql
-- イベント
create table events (
  id            text primary key,          -- nanoid(21)
  title         text not null,
  description   text,
  slot_type     text not null,             -- 'datetime' | 'timetable' | 'free'
  admin_pw_hash text not null,             -- bcrypt
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

-- 候補スロット(3形式を正規化して格納)
create table slots (
  id         bigint generated always as identity primary key,
  event_id   text not null references events(id) on delete cascade,
  label      text not null,                -- 表示ラベル(例：'7/20(月) 09:00')
  date_key   date,                         -- datetime形式のときのグリッド行列復元用
  time_key   time,                         -- 同上
  sort_order int not null
);

-- 回答者
create table respondents (
  id         bigint generated always as identity primary key,
  event_id   text not null references events(id) on delete cascade,
  name       text not null,                -- 同名可(別人扱い)
  pw_hash    text not null,                -- bcrypt(回答編集用)
  created_at timestamptz not null default now()
);

-- 回答(◯と△のみ保存．レコードなし＝×)
create table answers (
  respondent_id bigint not null references respondents(id) on delete cascade,
  slot_id       bigint not null references slots(id) on delete cascade,
  status        text not null check (status in ('ok', 'maybe')),  -- ok=◯, maybe=△
  primary key (respondent_id, slot_id)
);
```

### 6.3 設計メモ
- 未入力・×はレコードを持たない → 集計時は「回答者数 − (◯数+△数) = ×数」で導出
- `date_key`/`time_key`はdatetime形式のグリッド描画を高速化するための非正規化カラム(timetable/free形式ではnull)
- イベント削除時は外部キーCASCADEで全データが消える
- `updated_at`は回答の作成・編集・削除時にも更新する(自動削除の起点になるため)

## 7. API設計(Next.js API Routes)

| メソッド | パス | 内容 |
|---|---|---|
| POST | /api/events | イベント作成(管理pwをハッシュ化して保存) |
| GET | /api/events/[id] | イベント情報+スロット+集計データ取得 |
| POST | /api/events/[id]/slots | スロット追加(要：管理pw) |
| DELETE | /api/events/[id] | イベント削除(要：管理pw) |
| POST | /api/events/[id]/respondents | 新規回答(名前+pw+回答内容) |
| PUT | /api/respondents/[id] | 回答編集(要：本人pw) |
| DELETE | /api/respondents/[id] | 回答削除(要：本人pw または 管理pw) |

- パスワード検証は全てサーバー側で行う．検証失敗は401を返す
- レートリミット等は当面不要(サークル内利用のため)．荒らしが発生したら検討

## 9. 開発フェーズ

| Phase | 内容 | 完了条件 |
|---|---|---|
| 0 | リポジトリ整備 | Next.js雛形，CLAUDE.md，このspec.md配置，Vercelデプロイ疎通 |
| 1 | グリッドUIモック | DBなしで◯△×のモード切替+ドラッグ入力が実機(スマホ)で快適に動く |
| 2 | DB接続 | イベント作成→共有URL→回答保存→表示の一連が動く(datetime形式のみ) |
| 3 | 集計+主催者機能 | ヒートマップ，回答編集/削除，管理pw機能，mypage |
| 4 | 品質固め | Playwright E2E，LINE内ブラウザ実機検証，定期ジョブ設定 |
| 5 | β運用 | WASA班内で人柱テスト→フィードバック反映 |
| 6 | 拡張 | 時間割形式(P2)，自由ラベル形式(P3) |

- **Phase 1が最重要**．入力UXがダメなら全てがダメなので，Phase 1完了時点でWASAメンバー数人に触ってもらうこと
- 目標：Phase 5までを夏休み中(8月末)に完了

## 10. リスクと対策

| リスク | 対策 |
|---|---|
| ドラッグ入力がLINE内ブラウザで動かない | Phase 1で最優先検証．ダメならタップ+行列一括のみにフォールバック |
| Supabase無料枠の7日停止 | GitHub Actions毎日ping(4.5) |
| localStorage非対応環境 | mypage/名前自動入力をグレースフル無効化(2.7) |
| パスワード忘れ | 主催者削除→再回答のフローを画面上で案内 |
| 刻み幅自由入力による巨大グリッド | スロット数上限(例：500)をバリデーションで設ける |
